# Nathan Mike Sidi Bakari - Epic Breakdown

## Overview

This document translates the WDS design delivery and the current `develop` prototype into implementable epics and stories.

The current prototype is the candidate visual baseline: a minimal notebook/log archive with routes for chefs-d'oeuvre, timeline, lenses, notes, entry detail, about, contact, legal pages, RSS, and a search palette. The implementation goal is not to restart design; it is to productionize the chosen prototype while preserving its visual feel and archive-first intent.

## Requirements Inventory

### Functional Requirements

FR1: Home / Portal must explain the archive premise and expose the primary routes: chefs-d'oeuvre, timeline, lenses, notes, about, and contact.

FR2: Home / Portal must show a compact archive stats block with entries, notes, and years represented from real content data.

FR3: The global shell must provide consistent navigation, footer links, RSS access, and search access across public pages.

FR4: Entries must be represented with a typed content model including slug, title, date, type, status, summary, body, lenses, tags, optional role, outcome, links, related entries, and chef-d'oeuvre marker.

FR5: Notes must be represented with a typed content model including slug, title, date, summary, body, tags, reading time, and optional cover image metadata.

FR6: Chefs-d'oeuvre must list marked entries as curated timeline highlights, not as a conversion-oriented portfolio section.

FR7: Timeline must list all entries chronologically, grouped by year, with restrained chef-d'oeuvre emphasis and links into entry details.

FR8: Lenses must provide an index and individual lens pages for Engineering, Entrepreneurship, Investing, and Art, with overlap allowed across entries.

FR9: Entry Detail must render metadata, body, role, outcome, links, lenses, tags, related entries, previous/next navigation, and contextual back links.

FR10: Notes Index must list curated notes with dates, summaries, tags, reading time, and RSS access.

FR11: Note Detail must render long-form editorial reading pages with optional cover image, metadata, tags, previous/next navigation, and RSS access.

FR12: RSS must produce a valid XML feed for notes.

FR13: Search palette must allow users to quickly find entries and notes without overwhelming the minimal interface.

FR14: About, Contact, Colophon, Privacy, and Terms pages must be present, warm in tone, minimal, and consistent with the archive's non-conversion intent.

FR15: Public routes must expose appropriate page metadata, descriptions, Open Graph data where useful, and stable URLs.

FR16: Missing entry/note/lens routes must show helpful not-found states with return paths.

FR17: The prototype's notebook/log visual direction must be preserved while productionizing implementation details.

FR18: GitHub Pages or the selected free deployment target must be able to build and serve the site.

### NonFunctional Requirements

NFR1: The site must stay responsive across mobile and desktop.

NFR2: Navigation and primary actions must be keyboard accessible with visible focus states.

NFR3: The app must build without errors.

NFR4: Lint must pass with no blocking errors; existing component export warnings may be triaged separately if non-blocking.

NFR5: Internal route links must not be broken.

NFR6: The site must avoid conversion-first language, dominant CTAs, analytics pressure, or self-important copy.

NFR7: Static or mostly static content should load quickly.

NFR8: Content changes should be straightforward and safe for long-term maintenance.

### Additional Requirements

- Current app stack is TanStack Start / React / TypeScript / Tailwind, generated by Lovable and currently running on `develop`.
- The final production architecture remains a deliberate choice; for this backlog, treat the existing scaffold as the implementation surface unless a separate architecture decision changes it.
- Preserve WDS documentation and `.lovable/plan.md`; do not let implementation overwrite planning artifacts.
- Keep "lenses" as the current vocabulary for non-strict identity perspectives.
- Keep "chefs-d'oeuvre" as the current vocabulary for marked timeline entries unless copy review changes it.
- Search, legal pages, RSS, and notes are already present in the prototype and should be hardened rather than reinvented.
- GitHub Pages workflow currently needs real build/deploy steps once the stack is confirmed.

### UX Design Requirements

UX-DR1: Preserve the notebook/log aesthetic: minimal, precise, warm, artifact-led, and open to curious visitors.

UX-DR2: Maintain the compact stats/catalog block on the Home / Portal.

UX-DR3: Keep navigation restrained: chefs-d'oeuvre, timeline, lenses, notes, about, contact, and search.

UX-DR4: Treat chefs-d'oeuvre as marked entries from the timeline, not trophies.

UX-DR5: Treat lenses as ways of seeing the same archive, not strict categories.

UX-DR6: Keep contact copy warm and welcoming without turning into a sales funnel.

UX-DR7: Notes should feel like curated editorial writing, with comfortable reading pages and RSS.

UX-DR8: Entry pages should make metadata legible without becoming cluttered.

UX-DR9: Empty, not-found, and unavailable states should use plain, quiet copy.

UX-DR10: Footer should remain minimal and can include quiet legal/social/RSS links.

## Epic 1: Stabilize the Chosen Prototype Baseline

Goal: preserve the current `develop` prototype as the design baseline and make the global app shell dependable before deeper production work begins.

### Story 1.1: Capture Prototype Baseline

As the site owner,
I want the current good prototype captured as the implementation baseline,
So that future production work can improve it without losing the design direction.

**Acceptance Criteria:**

**Given** the current `develop` prototype is the preferred direction
**When** the baseline story is complete
**Then** the branch has a clean commit or tag documenting the chosen prototype state
**And** the commit message references the notebook/log design direction
**And** the WDS docs and `.lovable/plan.md` remain in the repository
**And** the app still runs and builds after the checkpoint

### Story 1.2: Verify App Shell and Primary Navigation

As a visitor,
I want a consistent shell across the archive,
So that I can move between major sections without losing context.

**Acceptance Criteria:**

**Given** I visit any public route
**When** I use the header or footer
**Then** I can navigate to chefs-d'oeuvre, timeline, lenses, notes, about, and contact
**And** the search trigger remains accessible without obscuring navigation
**And** active navigation styling is visible where supported
**And** no primary nav link points to a missing route

### Story 1.3: Establish Build and Lint Baseline

As a developer,
I want repeatable app checks,
So that future stories can be verified consistently.

**Acceptance Criteria:**

**Given** dependencies are installed
**When** I run the standard verification commands
**Then** the app builds successfully
**And** lint has no blocking errors
**And** known non-blocking warnings are documented if they remain
**And** generated route files are either committed or regenerated deterministically

## Epic 2: Home and Marked Entry Discovery

Goal: make the first impression and marked-entry path useful, warm, and faithful to the archive concept.

### Story 2.1: Refine Home Portal Messaging

As a first-time visitor,
I want the home page to explain the archive plainly,
So that I understand what this place is without feeling pitched to.

**Acceptance Criteria:**

**Given** I open the root route
**When** I read the premise section
**Then** the copy communicates a personal archive/logbook
**And** it avoids cold, defensive, or self-important phrasing
**And** it does not frame the site as a resume or conversion page
**And** it keeps the notebook/log visual rhythm from the prototype

### Story 2.2: Productionize Home Stats Block

As a visitor,
I want to see a compact summary of the archive,
So that the site feels tangible and browsable.

**Acceptance Criteria:**

**Given** entries and notes exist in content data
**When** I open the Home / Portal
**Then** the stats block displays entry count, note count, and year span from actual data
**And** counts update when content data changes
**And** the block remains readable on mobile and desktop
**And** empty content does not break the layout

### Story 2.3: Harden Chefs-d'oeuvre Entry Path

As a curious visitor,
I want a short path into marked entries,
So that I can get a strong first impression without browsing the whole archive.

**Acceptance Criteria:**

**Given** entries can be marked as chefs-d'oeuvre
**When** I open the chefs-d'oeuvre route
**Then** only marked entries appear
**And** each listed entry links to its detail page
**And** the page copy presents marked entries as entry points, not trophies
**And** the page handles no marked entries with quiet empty-state copy

## Epic 3: Entry Model and Entry Detail

Goal: make entries the durable core of the archive.

### Story 3.1: Validate Entry Content Model

As the site owner,
I want entries to have a reliable content model,
So that timeline, lenses, search, and details all derive from the same data.

**Acceptance Criteria:**

**Given** the app has an `Entry` model
**When** entries are loaded
**Then** required fields are present for slug, title, date, type, status, summary, body, lenses, and tags
**And** optional fields for role, outcome, links, related entries, and chef-d'oeuvre are supported
**And** invalid lenses or duplicate slugs are caught during development or build checks
**And** entries can belong to multiple lenses

### Story 3.2: Productionize Entry Detail Page

As a visitor,
I want to inspect one entry with context and metadata,
So that I understand what happened and how it connects.

**Acceptance Criteria:**

**Given** an entry exists
**When** I open `/entries/$slug`
**Then** the page displays title, date, type, status, summary, body, lenses, tags, role/outcome when present, links, and related entries when present
**And** the page includes SEO metadata derived from the entry
**And** related-entry links resolve correctly
**And** the page remains readable on mobile and desktop

### Story 3.3: Handle Missing and Sparse Entries

As a visitor,
I want missing or sparse entries to fail gracefully,
So that browsing does not feel broken.

**Acceptance Criteria:**

**Given** I open an unknown entry slug
**When** the route cannot find the entry
**Then** I see a quiet not-found state with a return path
**And** valid entries with no media still render body and metadata correctly
**And** entries with no related items omit the related section or show a quiet fallback
**And** no page crashes due to optional entry fields

## Epic 4: Timeline, Lenses, and Search Discovery

Goal: make archive discovery work through time, perspective, and quick lookup.

### Story 4.1: Harden Timeline Browsing

As Future Nathan or a returning visitor,
I want to browse entries by date,
So that trajectory and memory are visible.

**Acceptance Criteria:**

**Given** entries have dates
**When** I open Timeline
**Then** entries are grouped or ordered by year/date with newest first
**And** chefs-d'oeuvre receive restrained visual emphasis
**And** each entry links to the correct detail page with timeline context
**And** the timeline handles empty entry data without breaking

### Story 4.2: Harden Lens Browsing

As a visitor,
I want to browse the same archive through different lenses,
So that overlap across domains feels natural.

**Acceptance Criteria:**

**Given** entries have one or more lenses
**When** I open Lenses
**Then** I see Engineering, Entrepreneurship, Investing, and Art with counts and preview entries
**And** each lens detail page lists matching entries
**And** entries with multiple lenses appear in each relevant lens
**And** invalid lens slugs show a helpful not-found state

### Story 4.3: Stabilize Search Palette

As a visitor,
I want quick search access,
So that I can find entries or notes without navigating every route.

**Acceptance Criteria:**

**Given** entries and notes exist
**When** I open the search palette
**Then** I can find entries and notes by title, summary, tags, or lens where supported
**And** selecting a result navigates to the correct route
**And** keyboard interaction works for opening, moving through results, selecting, and closing
**And** empty search results are communicated with plain copy

## Epic 5: Notes and RSS Reading Experience

Goal: make editorial notes feel durable, readable, and followable.

### Story 5.1: Harden Notes Index

As a reader,
I want to browse notes by date and summary,
So that I can choose what to read without a noisy feed.

**Acceptance Criteria:**

**Given** notes exist
**When** I open Notes
**Then** notes are listed by date with title, summary, reading time, and tags
**And** the RSS link is visible but not pushy
**And** the layout remains readable on mobile and desktop
**And** the empty state uses quiet copy if no notes exist

### Story 5.2: Productionize Note Detail

As a reader,
I want a comfortable long-form note page,
So that I can read without distraction.

**Acceptance Criteria:**

**Given** a note exists
**When** I open `/notes/$slug`
**Then** the page renders title, date, reading time, summary, body, tags, optional cover image, and previous/next note links
**And** SEO metadata is derived from the note
**And** the reading column is comfortable on desktop and mobile
**And** unknown note slugs show a quiet not-found state

### Story 5.3: Validate RSS Feed

As a reader who uses RSS,
I want a valid notes feed,
So that I can follow updates without a platform feed.

**Acceptance Criteria:**

**Given** notes exist
**When** I open `/rss.xml`
**Then** the response is valid RSS XML with note title, link, guid, pubDate, and description
**And** item links point to valid note routes
**And** XML escaping handles special characters
**And** RSS is linked from root metadata and notes pages

## Epic 6: Production Readiness, Copy Tone, SEO, and Deployment

Goal: make the selected design ready to release without losing warmth or maintainability.

### Story 6.1: Copy Tone Review

As the site owner,
I want copy that is warm, precise, and unforced,
So that the archive feels open rather than pompous or cold.

**Acceptance Criteria:**

**Given** all public route copy is available
**When** copy review is complete
**Then** contact, about, notes, chefs-d'oeuvre, lenses, and homepage copy avoid defensive "not X" framing unless useful
**And** contact copy welcomes genuine overlap without sounding unavailable
**And** the archive still avoids conversion-oriented language
**And** any final vocabulary decisions are consistent across nav, routes, metadata, and page copy

### Story 6.2: SEO and Social Metadata Pass

As a visitor arriving from search or shared links,
I want pages to have accurate metadata,
So that previews and search snippets match the site.

**Acceptance Criteria:**

**Given** public routes are implemented
**When** metadata review is complete
**Then** root, chefs-d'oeuvre, timeline, lenses, lens detail, entry detail, notes, note detail, about, contact, colophon, privacy, and terms have appropriate titles and descriptions
**And** entry and note pages derive metadata from content data
**And** RSS alternate links are present where appropriate
**And** metadata does not overstate the site's purpose

### Story 6.3: Accessibility and Responsive QA

As a visitor on any device,
I want the site to be readable and navigable,
So that the archive is usable without special conditions.

**Acceptance Criteria:**

**Given** the main route set is implemented
**When** QA is complete
**Then** primary routes are keyboard accessible
**And** visible focus states exist for interactive elements
**And** heading order is logical
**And** mobile layouts preserve readability and navigation
**And** images have useful alt text where images are meaningful

### Story 6.4: GitHub Pages Deployment

As the site owner,
I want the site deployed from the repository,
So that the archive is publicly accessible.

**Acceptance Criteria:**

**Given** the app stack is confirmed for release
**When** deployment setup is complete
**Then** the GitHub Pages workflow installs dependencies, builds the app, uploads the correct output directory, and deploys successfully
**And** the workflow uses a Node version compatible with the app dependencies
**And** route fallback behavior is configured for the selected build output
**And** deployment instructions or known limits are documented
